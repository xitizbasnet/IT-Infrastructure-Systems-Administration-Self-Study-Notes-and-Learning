# 4.1 LDAP — Lightweight Directory Access Protocol

## 🎯 Objective

Configure and validate LDAP connectivity for directory-based authentication and identity queries.

---

# 🛠️ Step 1: Test LDAP Connectivity

## Execute LDAP Query

```bash
ldapsearch -x -H ldap://dc01.corp.local -b 'DC=corp,DC=local' -D 'CN=svc_ldap,OU=ServiceAccts,DC=corp,DC=local' -W '(objectClass=user)'
```

---

## Parameter Breakdown

| Parameter | Description                  |
| --------- | ---------------------------- |
| `-x`      | Simple authentication        |
| `-H`      | LDAP server URI              |
| `-b`      | Base Distinguished Name (DN) |
| `-D`      | Bind DN (service account)    |
| `-W`      | Prompt for password          |

---

## Example Components

| Component       | Example                  |
| --------------- | ------------------------ |
| LDAP Server     | `ldap://dc01.corp.local` |
| Base DN         | `DC=corp,DC=local`       |
| Service Account | `svc_ldap`               |

---

# 🛠️ Step 2: Secure LDAP (LDAPS) — Enable on AD

## Install SSL Certificate on Domain Controller

Install a certificate from:

* Internal Certificate Authority (CA)
* Self-signed certificate (testing environments only)

---

## Verify LDAPS Connectivity

### Open LDP Utility

Run:

```text id="9r4r7k"
ldp.exe
```

### Connect Using SSL

Navigate to:

```text id="ajc0yk"
Connection → Connect
```

Configure:

| Setting | Value   |
| ------- | ------- |
| Port    | `636`   |
| SSL     | Checked |

> 🔒 LDAPS encrypts LDAP traffic and protects authentication credentials in transit.

# 4.2 Kerberos Authentication Flow

## 🎯 Objective

Validate Kerberos authentication and troubleshoot common ticketing issues.

---

# 🛠️ Step 1: Verify Kerberos Tickets (Windows)

## Display Active Kerberos Tickets

```powershell id="x6j4b0"
klist
```

### Typical Information Displayed

* Ticket Granting Ticket (TGT)
* Service Tickets
* Expiration Times
* Encryption Types

---

# 🛠️ Step 2: Troubleshoot Kerberos

## Common Kerberos Errors

| Error                    | Description                                  |
| ------------------------ | -------------------------------------------- |
| `KDC_ERR_PREAUTH_FAILED` | Incorrect password or authentication failure |
| `KRB_AP_ERR_SKEW`        | System clock skew exceeds 5 minutes          |

---

## Time Synchronization Requirement

> ⏱️ Kerberos requires closely synchronized system clocks.

Recommended synchronization method:

```text id="6bznm0"
NTP (Network Time Protocol)
```

---

## Verify FSMO Roles

Run:

```powershell id="6h6k0d"
netdom query fsmo
```

> 🔍 Used to verify the Key Distribution Center (KDC) and FSMO role holders.

# 4.3 NTLM (Legacy — Use Only Where Required)

## 🎯 Objective

Audit and minimize the use of legacy NTLM authentication.

> ⚠️ NTLM should only be used where Kerberos is not supported.

---

# 🛠️ Step 1: Audit NTLM Usage

## Enable NTLM Auditing via Group Policy

Navigate to:

```text id="dj2k7v"
Computer Configuration
→ Windows Settings
→ Security Settings
→ Local Policies
→ Security Options
```

Configure the following policy:

```text id="7qgrqo"
Network Security: Restrict NTLM: Audit NTLM authentication in this domain = Enable all
```

> 📋 Auditing assists in identifying systems and applications still dependent on NTLM.



Set the following values from the Service Provider (SP) metadata:

| Setting             | Description                           |

# 🛠️ Step 2: OIDC Token Request (PowerShell)

## Define Request Body

```powershell id="1vvz74"
$body = @{
    grant_type='client_credentials'
    client_id=''
    client_secret=''
    scope='https://graph.microsoft.com/.default'
}
```

---

## Request Access Token

```powershell id="hty9z7"
Invoke-RestMethod -Method POST -Uri 'https://login.microsoftonline.com//oauth2/v2.0/token' -Body $body
```

> 🔑 The returned access token can be used to authenticate API requests.

# ✅ Best Practice Tips

## 🔒 Authentication Security Best Practices

* Always prefer Kerberos over NTLM authentication.
* Block NTLMv1 using Group Policy Objects (GPO).
* Use LDAPS exclusively on:

```text id="h4jdfv"
Port 636
```

* Disable plain LDAP (`389`) wherever operationally feasible.

---

## 🛡️ Federation & SSO Security Recommendations

### SAML Recommendations

* Rotate SAML signing certificates annually.
* Update Service Provider metadata after certificate changes.
* Validate NameID and claims mappings during testing.

### OIDC Recommendations

* Implement token lifetime policies to reduce session risk.
* Use short-lived access tokens where possible.
* Store secrets securely using managed identity or vault solutions.

---

## ☁️ Microsoft Entra ID Recommendations

* Enable Conditional Access policies before deploying SSO.
* Require Multi-Factor Authentication (MFA) for privileged roles.
* Audit sign-in logs regularly for anomalous behavior.

---

## 📊 Monitoring & Operational Recommendations

* Enable authentication auditing across all identity systems.
* Synchronize all systems with reliable NTP sources.
* Regularly review legacy authentication usage reports.
* Maintain backup administrative accounts for federation outages.
